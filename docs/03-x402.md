# 03 · x402 — the client-side payment handshake

**Goal:** call a resource that requires payment, pay it programmatically, and
get the unlocked response — the loop most builders get wrong.

**Files:** [`src/lib/x402.ts`](../src/lib/x402.ts) (the clearest file in the
repo, by design), [`src/components/X402Fetch.tsx`](../src/components/X402Fetch.tsx),
[`src/app/api/x402-demo/route.ts`](../src/app/api/x402-demo/route.ts)

## The protocol in four steps

x402 revives HTTP `402 Payment Required` as a real protocol:

1. **Request** the resource with no payment.
2. **402 challenge** — the server replies `402` with a JSON body describing
   what to pay, to whom, on which chain, in which asset. *This is data, not an
   error.*
3. **Sign** a payment authorization. We use **EIP-3009
   `TransferWithAuthorization`** over USDC — a gasless transfer that a
   facilitator (not the signer) can later submit. Base64-encode it into an
   `X-PAYMENT` header.
4. **Retry** the same request with that header. The server verifies the
   signature, settles it (via a facilitator, in production), and returns the
   payload.

Arke proved this with the feed at `feed.arke.live:8402`. This kit ships a local
402 endpoint so you can watch the whole handshake with no external services.

## What most builders get wrong

- **Treating 402 as a failure.** It's an expected step in a retry loop. Parse
  the body and continue.
- **Signing the wrong EIP-712 domain.** If `chainId` or the token address
  (`verifyingContract`) is off by anything, the recovered signer won't match
  and verification fails silently. The fix in this kit:
  `buildTransferAuthorization` is a **single pure function used by both sides**,
  so the client and server can't drift.
- **Forgetting it's gasless.** The signer doesn't pay gas; a facilitator
  submits the EIP-3009 transfer. That's what lets an agent holding only USDC
  pay for things.

## EIP-3009 `TransferWithAuthorization`

```
domain = { name: "USDC", version: "2", chainId, verifyingContract: <USDC token> }
types.TransferWithAuthorization = [
  from, to, value, validAfter, validBefore, nonce(bytes32)
]
```

The `nonce` is a fresh random 32 bytes per authorization — it's what prevents a
captured payment from being replayed.

## What the demo does and doesn't do

The demo server (`/api/x402-demo`) **verifies the signature offline**: it
rebuilds the typed data and recovers the signer with viem's `verifyTypedData`,
then checks recipient, amount, and expiry. It deliberately **does not settle
onchain** — settlement is a separate, swappable step that, in production, a
facilitator performs before returning the payload. Keeping it offline is what
lets this kit run with zero external dependencies. The handshake *shape* is
identical to a real x402 server.

## Fork just this

Take `src/lib/x402.ts` and the API route. The hook (`useX402Fetch`) works
against any x402 server; point it at a real endpoint and swap the demo
verification for a facilitator call.
